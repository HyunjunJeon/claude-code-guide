Rewind는 단순한 undo가 아니라 “어느 시점으로 돌아가 어떤 정보를 남길지”를 선택하는 작업입니다. 같은 checkpoint라도 코드와 대화를 모두 되돌릴지, 대화만 되돌릴지, 코드만 되돌릴지에 따라 결과가 달라집니다.

## 선택 기준

| 상황 | 권장 선택 |
|---|---|
| Claude가 잘못된 코드 변경을 시작함 | 코드와 대화를 함께 복원 |
| 코드 변경은 괜찮지만 대화 방향만 꼬임 | 대화만 복원 |
| 대화 맥락은 유지하고 파일만 이전 상태로 되돌리고 싶음 | 코드만 복원 |
| 실패한 접근을 다음 시도에 반영하고 싶음 | 선택 지점부터 요약 |

## 실전 패턴

1. 큰 변경 전에 “이 상태를 기준점으로 삼자”고 명시합니다.
2. Claude가 여러 파일을 바꾸게 두되, 중간마다 테스트나 diff를 확인합니다.
3. 결과가 틀렸다면 `Esc` + `Esc` 또는 `/rewind`로 checkpoint 목록을 엽니다.
4. 되돌린 뒤 실패 원인을 짧게 정리하고 더 작은 범위로 다시 요청합니다.

## 좋은 요청 예시

```text
이 리팩터링을 시작하기 전 상태로 돌아가자. 코드와 대화를 모두 되돌리고, 실패 원인만 세 줄로 요약해줘.
```

```text
현재 파일 변경은 유지하고, 대화만 이전 설계 검토 시점으로 되돌려서 다른 구현 계획을 세우자.
```
