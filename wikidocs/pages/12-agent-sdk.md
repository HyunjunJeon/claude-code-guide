# 12. Agent SDK

Claude Code Agent SDK는 Claude의 에이전트 루프를 본인의 앱이나 워크플로에 코드 수준에서 임베드할 수 있게 해주는 라이브러리입니다.  
대화형 CLI 사용자가 아니라, TypeScript 또는 Python으로 세션, 도구, 권한, 스트리밍, 호스팅을 직접 제어해야 하는 빌더 관점의 모듈입니다.

## 언제 읽으면 좋은가

- Claude Code의 자동화 흐름을 본인의 서비스나 사내 도구에 임베드하고 싶을 때
- TypeScript 또는 Python으로 에이전트 루프와 도구 호출을 직접 제어해야 할 때
- 호스팅, 권한, 세션 영속성, 비용 추적을 운영 코드로 관리해야 할 때
- CLI 명령에서 가능한 동작을 백엔드 API나 백그라운드 작업으로 옮기고 싶을 때

## 핵심 개념

| 개념 | 무엇인가 | 먼저 볼 페이지 |
| --- | --- | --- |
| Agent loop | LLM 호출과 도구 실행을 묶는 에이전트의 기본 실행 루프 | `agent-loop.md` |
| Sessions | 대화 상태와 컨텍스트의 영속 단위 | `sessions.md` |
| Custom tools | 자체 함수를 도구로 노출해 에이전트가 호출하게 하는 메커니즘 | `custom-tools.md` |
| Permissions | 도구 사용과 파일 접근을 코드로 통제하는 정책 계층 | `permissions.md` |
| Hooks | SDK 실행 흐름의 특정 시점을 가로채 사용자 코드를 실행 | `hooks.md` |
| Streaming | 토큰과 도구 결과를 실시간으로 받아 UI에 흘려보내기 | `streaming-output.md` |
| Hosting | SDK 기반 에이전트를 운영 환경에 띄우는 패턴 | `hosting.md` |

## 빠른 시작

가장 짧은 진입은 TypeScript 또는 Python에서 SDK 클라이언트를 만들고 한 번의 에이전트 호출을 실행해보는 것입니다.

```ts
// TypeScript 예시 (자세한 내용은 typescript.md)
import { Claude } from "@anthropic-ai/claude-code-sdk";

const client = new Claude({ apiKey: process.env.ANTHROPIC_API_KEY });
const session = await client.sessions.create();
const result = await session.run({ input: "이 저장소의 README를 요약해줘" });
console.log(result.output);
```

```python
# Python 예시 (자세한 내용은 python.md)
from claude_code_sdk import Claude

client = Claude(api_key=os.environ["ANTHROPIC_API_KEY"])
session = client.sessions.create()
result = session.run(input="이 저장소의 README를 요약해줘")
print(result.output)
```

## 자주 하는 실수

- CLI에서 가능한 동작이 SDK에서도 자동으로 가능하다고 가정하기 — 일부 기능은 SDK 전용 또는 CLI 전용이라 `migration-guide.md` 확인이 필수
- `permissions` 설정 없이 운영 환경 배포 — 기본 정책은 안전하지 않으므로 명시적 정책 정의 필요
- 세션 ID를 매 요청마다 새로 만들기 — 비용과 컨텍스트 효율을 위해 세션 재사용 패턴이 기본
- Streaming 모드를 채택했는데 클라이언트가 partial 결과를 처리할 준비가 안 된 채 띄우기

## 12-01. 에이전트 루프

에이전트 루프는 Agent SDK 런타임의 중심입니다. 프롬프트를 받아 도구 결정, 도구 실행, 후속 추론, 최종 응답으로 이어지는 반복 사이클을 뜻합니다.

### 무엇인가

큰 흐름은 다음과 같습니다.

1. 앱이 프롬프트와 실행 옵션을 보냅니다.
2. Claude가 바로 답할지, 도구를 요청할지 결정합니다.
3. 호스트가 도구를 실행하고 결과를 다시 전달합니다.
4. Claude가 최종 답, 중단 조건, 턴 제한에 도달할 때까지 이 과정을 반복합니다.

권한과 훅은 이 루프 바깥에서 요청을 차단하거나 수정하거나 기록합니다.

### 언제 신경 써야 하나

다음 상황에서 루프 이해가 중요합니다.

- 에이전트가 왜 계속 도구를 호출하는지 디버깅할 때
- `maxTurns`로 과도한 반복을 막고 싶을 때
- 스트리밍 모드와 단일 실행 모드의 차이를 설명해야 할 때
- 정책과 로깅을 어느 경계에 넣어야 할지 결정할 때

### 멘탈 모델

SDK는 "프롬프트 입력, 답변 출력" 한 번으로 끝나는 래퍼가 아닙니다. 모델 단계와 도구 단계가 반복되는 이벤트 루프입니다. 그래서 세션, 훅, 권한, 관측성이 핵심이 됩니다.

### 제어 지점

- `query(...)`가 루프를 시작합니다.
- `maxTurns`가 반복 길이를 제한합니다.
- `allowedTools`, `disallowedTools`가 요청 가능한 도구를 좁힙니다.
- `permissionMode`, 선언형 설정, `canUseTool`이 승인 여부를 결정합니다.
- `PreToolUse`, `PostToolUse` 같은 훅이 실행 전후 동작을 관찰하거나 수정합니다.

### 흔한 실수

- 반복 도구 호출을 무조건 버그로 보는 것
- 차단된 도구 요청도 대화 흐름에는 영향을 준다는 점을 놓치는 것
- 승인이나 후속 입력이 필요한데도 단일 실행 모드를 택하는 것
- 세션이 루프 제어까지 대신한다고 생각하는 것

## 12-02. 비용 추적

비용 추적은 Agent SDK 실행이 얼마나 많은 비용을 쓰는지, 그리고 그 지출이 어디서 발생하는지를 파악하는 작업입니다.

### 무엇을 측정할까

유용한 측정 항목은 다음과 같습니다.

- 전체 실행 비용
- 토큰 사용량
- 도구 중심 실행과 모델 중심 실행의 차이
- 워크플로, 사용자, 테넌트별 비용
- 재시도나 긴 컨텍스트 체인 때문에 생기는 급증

### 왜 중요한가

에이전트 앱은 정적인 프롬프트 래퍼가 아닙니다. 사용자 행동 하나가 여러 모델 턴과 도구 호출로 확장될 수 있습니다. 비용 추적이 없으면 어떤 워크플로가 효율적인지, 단지 비싼지만 알기 어렵습니다.

### 실전 패턴

- 세션 기록에 사용량과 비용 메타데이터를 붙입니다.
- 운영자에게 실행별 요약을 보여줍니다.
- 비정상적인 사용량 증가에 경보를 겁니다.
- `maxTurns`, 권한 정책과 함께 봅니다.
- 평가 트래픽과 실제 운영 트래픽을 구분합니다.

### 흔한 실수

- 총액만 보고 어떤 워크플로가 원인인지 놓치는 것
- 재시도, 도구 루프, 컨텍스트 비대화를 무시하는 것
- 테스트/프리뷰 트래픽과 실제 사용자 트래픽을 섞는 것

### 관련 링크

- [관측성](12-agent-sdk.md#12-agent-sdk-10-관측성)
- [세션](12-agent-sdk.md#12-agent-sdk-16-세션)

## 12-03. 커스텀 도구

커스텀 도구는 Agent SDK에 프로젝트 전용 기능을 붙이는 방법입니다. 보통 인프로세스 MCP 서버로 정의해서 Claude가 기본 도구처럼 호출하게 만듭니다.

### 무엇인가

도구의 스키마와 핸들러를 정의한 뒤, MCP 서버로 등록해서 `query(...)`에 넘깁니다. Claude는 빌트인 도구와 함께 이 도구를 볼 수 있지만, 허용한 것만 사용할 수 있습니다.

### 언제 쓰는가

내장된 파일/셸/웹 도구만으로 부족할 때 씁니다. 예를 들면 사내 API 조회, 데이터베이스 검색, 브라우저 자동화, 계산기, 서비스별 워크플로가 있습니다.

### 멘탈 모델

흐름은 세 단계로 보면 됩니다.

1. 도구 계약과 핸들러를 정의한다.
2. 도구 묶음을 MCP 서버로 등록한다.
3. Claude가 사용할 도구만 명시적으로 허용한다.

이렇게 해야 확장성은 유지하면서 접근 범위는 좁게 가져갈 수 있습니다.

### 핵심 API와 패턴

- `createSdkMcpServer(...)`로 인프로세스 MCP 서버를 만듭니다.
- `tool(...)`로 이름, 설명, 스키마, 핸들러를 정의합니다.
- `mcpServers`로 서버를 `query(...)`에 전달합니다.
- `allowedTools`에는 필요한 `mcp__{server_name}__{tool_name}`만 넣습니다.
- MCP 도구를 쓸 때는 문자열 프롬프트보다 async generator/iterable을 쓰는 스트리밍 입력 모드가 필요합니다.
- `maxTurns`를 두면 커스텀 도구가 여러 턴을 유발해도 제어하기 쉽습니다.

이름 규칙 예시:

- `get_weather`를 `my-custom-tools` 서버에 두면 `mcp__my-custom-tools__get_weather`가 됩니다.

### 흔한 실수

- MCP 서버를 쓰면서 문자열 프롬프트를 그대로 넣는 것
- 서버 전체를 열어 놓고 세부 도구를 제한하지 않는 것
- 입력과 출력을 검증하지 않는 것
- 에러 메시지를 충분히 의미 있게 돌려주지 않는 것

## 12-04. 파일 체크포인팅

파일 체크포인팅은 안전한 에이전트 실행을 위한 파일 상태 측면의 장치입니다. 편집 전에 롤백 지점을 만들고, 실행이 잘못 흘러가면 파일을 되돌릴 수 있게 합니다.

### 세션과 어떻게 다른가

- 세션은 대화 기록을 보존합니다.
- 파일 체크포인트는 작업 공간 상태를 보존합니다.

둘은 연결될 수 있지만 서로 대체하지는 못합니다. 세션을 이어도 파일은 되돌아오지 않고, 체크포인트를 복원해도 Claude의 추론 기록이 복원되지는 않습니다.

### 언제 쓰는가

다음 상황에서 특히 유용합니다.

- 되돌릴 수 있는 편집 워크플로
- 장시간 리팩터링
- 실패 가능성이 있는 실험성 수정
- 자동 편집에 대한 운영자 신뢰 확보

### 운영 가이드

- 넓은 범위의 편집 전에는 먼저 체크포인트를 만듭니다.
- 이름이나 태그를 붙여서 호스트가 명확히 보여줄 수 있게 합니다.
- 감사 가능성이 중요하면 세션 ID와 함께 관리합니다.
- 롤백은 부가 기능이 아니라 1급 UI 동작으로 다룹니다.

### 흔한 실수

- 이미 파일을 바꾼 뒤에 체크포인트를 찍는 것
- Git만 있으면 애플리케이션 롤백 UX가 충분하다고 보는 것
- 네트워크 호출 같은 외부 부작용도 되돌아간다고 착각하는 것

<a id="12-agent-sdk-05-훅"></a>

## 12-05. 훅

훅은 Agent SDK의 라이프사이클 주변에서 코드를 실행하는 방법입니다. 감사 로그, 검증, 정책 집행, 컨텍스트 주입, 자동 후처리에 적합합니다.

### 무엇인가

훅은 Claude가 도구를 부르기 전, 호출한 뒤, 세션이 시작/종료될 때, 사용자가 프롬프트를 제출할 때처럼 특정 경계에서 실행되는 이벤트 기반 콜백 또는 명령입니다.

### 언제 쓰는가

다음이 필요할 때 사용합니다.

- 도구 사용 내역을 기록하거나 감사하고 싶을 때,
- 위험한 작업을 차단하고 싶을 때,
- Claude가 답하기 전에 추가 컨텍스트를 넣고 싶을 때,
- 출력 형식을 정리하거나 후처리하고 싶을 때,
- 작업 디렉터리 변경이나 파일 변화를 감시하고 싶을 때.

### 멘탈 모델

훅은 루프 자체가 아니라 루프를 둘러싼 제어층입니다. `PreToolUse`는 실행 전에 요청을 차단하거나 수정할 수 있고, `PostToolUse`는 도구가 이미 끝난 뒤 결과를 봅니다. 세션 훅과 프롬프트 훅은 라이프사이클 경계에서 동작합니다.

### 핵심 API와 패턴

- `settings.json`, `.claude/settings.local.json`, 사용자/프로젝트 설정에서 훅을 구성합니다.
- matcher로 도구 이름 범위를 좁힐 수 있습니다.
- 주요 이벤트는 `PreToolUse`, `PostToolUse`, `SessionStart`, `SessionEnd`, `UserPromptSubmit`, `Notification`, `Stop`, `FileChanged`, `CwdChanged`입니다.
- 훅 출력은 exit code 또는 `permissionDecision`, `permissionDecisionReason`, `updatedInput`, `additionalContext`, `updatedMCPToolOutput` 같은 JSON 필드를 사용합니다.
- `CLAUDE_PROJECT_DIR`를 쓰면 현재 디렉터리에 의존하지 않고 프로젝트 스크립트를 안전하게 참조할 수 있습니다.

### 흔한 실수

- `PostToolUse`에서 도구 실행 자체를 막을 수 있다고 생각하는 것
- 변수 인용과 절대 경로를 빼먹은 취약한 셸 명령을 쓰는 것
- 설정 변경이 현재 세션에 즉시 반영된다고 기대하는 것
- 훅을 권한 설계의 대체물로 보는 것
- 훅이 임의의 셸 명령을 실행한다는 보안 전제를 가볍게 보는 것

## 12-06. 호스팅

호스팅은 Agent SDK 애플리케이션을 어디에서 실행하고, 어떤 인프라가 그 주위를 둘러싸는지를 결정하는 문제입니다.

### 흔한 배포 형태

대표적인 선택지는 다음과 같습니다.

- 로컬 개발자 머신
- CI 러너
- 컨테이너 기반 웹 서비스
- 사내 자동화 워커
- 테넌트 분리를 고려한 백엔드 서비스

정답은 도구 접근 범위, 격리 수준, 지연 허용치, 누가 실행을 트리거하는지에 따라 달라집니다.

### 초기에 정해야 할 것

- 세션을 어디에 저장할지
- 인증 정보를 어떤 방식으로 공급할지
- 도구가 파일 시스템이나 셸에 접근할 수 있는지
- 실행이 얼마나 오래 살아도 되는지
- 로그와 추적을 어떻게 수집할지

이 결정들이 앱 전체의 안전 모델을 좌우합니다.

### 흔한 실수

- 의도보다 넓은 파일 시스템 접근 권한이 있는 위치에 에이전트를 올리는 것
- 장시간 세션에 내구성 있는 저장소가 필요하다는 점을 놓치는 것
- 개발 환경 호스팅과 운영 호스팅을 같은 것으로 보는 것

<a id="12-agent-sdk-07-mcp"></a>

## 12-07. MCP

MCP(Model Context Protocol)는 외부 도구, 리소스, 프롬프트를 에이전트에 붙이는 표준 확장 방식입니다. 모든 연동을 호스트 앱 코드에 직접 박아 넣지 않고도 기능을 열 수 있게 해 줍니다.

### 무엇을 더해 주는가

MCP 서버는 보통 다음을 노출합니다.

- 에이전트가 호출할 수 있는 도구
- 에이전트가 읽을 수 있는 리소스
- 호스트가 노출할 수 있는 프롬프트나 템플릿

그래서 MCP는 사내 API, 개발 도구, 외부 시스템, 앱 커넥터를 연결하는 표준 브리지 역할을 합니다.

### SDK에서 왜 중요한가

Agent SDK에서 MCP는 "빌트인 도구만 쓴다"에서 "우리 플랫폼의 실제 시스템과 연결한다"로 넘어가는 핵심 경계입니다. 또한 모든 기능을 하나의 거대한 호스트 서비스에 집어넣는 방식보다 훨씬 정돈된 확장 모델을 제공합니다.

### 설계 가이드

- 워크플로에 필요한 서버와 도구만 노출합니다.
- 가능하면 인증은 모델 밖에서 처리합니다.
- 서버 출력은 신뢰하기 전에 한 번 더 검증합니다.
- 운영자를 위해 도구 이름과 실패 방식을 문서화합니다.

간단한 인프로세스 확장이라면 SDK 커스텀 도구를 MCP 서버처럼 감싸서 동일한 인터페이스로 보이게 할 수도 있습니다.

### 흔한 실수

- 큰 서버를 연결해 놓고 `allowedTools`를 좁히지 않는 것
- MCP 인증과 최종 사용자 인증을 같은 것으로 보는 것
- 내부 서버에서 왔다는 이유만으로 리소스를 안전하다고 간주하는 것

## 12-08. 마이그레이션 가이드

마이그레이션 가이드는 예전 Claude Code SDK 사용 패턴에서 현재 Claude Agent SDK 모델로 옮기는 팀을 위한 문서입니다.

4. 넓게 배포하기 전에 관측성을 먼저 붙입니다.

## 12-09. 시스템 프롬프트 수정

시스템 프롬프트 수정은 사용자 메시지가 들어오기 전에 에이전트의 상시 동작을 바꾸는 작업입니다.

### 언제 쓰는가

다음처럼 오래 유지되어야 하는 동작이 필요할 때 사용합니다.

- 팀 스타일이나 출력 제약
- 도메인 특화 지시사항
- 컴플라이언스 문구
- 호스트 애플리케이션 전용 상호작용 규칙

다만 과도하게 키우면 디버깅이 어렵습니다.

### 좋은 방식

- 전체 교체보다 작은 지시를 덧붙이는 방식을 우선합니다.
- 정책성 지시와 작업별 지시는 분리합니다.
- 프롬프트 변경도 코드 변경처럼 기록하고 리뷰합니다.
- 단순 예제가 아니라 실제 도구 사용 시나리오로 검증합니다.

### 피해야 할 것

- 권한이나 훅으로 강제해야 할 정책을 프롬프트 문장에만 숨기는 것
- 자주 바뀌는 작업 세부사항을 기본 시스템 프롬프트에 넣는 것
- 기본 프롬프트가 제공하던 제품 동작을 이해하지 않은 채 통째로 교체하는 것

## 12-10. 관측성

관측성은 운영 환경에서 에이전트가 실제로 무엇을 하는지 보는 체계입니다. 어떤 프롬프트가 들어왔고, 어떤 도구가 호출됐고, 왜 실패했는지, 지연과 비용이 어떻게 나왔는지를 설명해 줍니다.

### 좋은 관측성이 담아야 할 것

최소한 다음은 남겨야 합니다.

- 세션 ID와 사용자/요청 상관관계
- 모델 선택과 실행 옵션
- 도구 요청과 결과
- 승인 또는 거절 결정
- 지연, 토큰 사용량, 최종 상태

규모가 커지면 테넌트 경계, 환경, 배포 버전도 함께 기록하는 편이 좋습니다.

### 왜 중요한가

에이전트가 실패했을 때 "답이 이상했다"만으로는 부족합니다. 권한, 도구, 누락된 컨텍스트, 스키마 검증, 모델 동작 중 어디서 문제가 생겼는지 분해할 수 있어야 합니다.

### 실전 패턴

- 세션 ID를 추적의 중심 키로 씁니다.
- 도구 이벤트는 사용자용 텍스트와 분리해서 기록합니다.
- 외부로 내보내기 전에 비밀 정보를 마스킹합니다.
- 비용 추적과 경보 체계에 연결합니다.
- 정책이 허용하는 범위 안에서만 충분한 디버깅 데이터를 남깁니다.

### 흔한 실수

- 마스킹 계획 없이 전부 로그에 남기는 것
- 사용자에게 보이는 텍스트만 저장하고 도구 이력을 놓치는 것
- 재시도나 세션 포크를 하나의 흐름으로 묶지 못하는 것

<a id="12-agent-sdk-11-개요"></a>

## 12-11. 개요

Claude Agent SDK는 Claude Code를 애플리케이션 안에서 프로그래밍 가능하게 쓸 수 있도록 만든 SDK입니다. 파일 읽기, 도구 호출, 멀티턴 컨텍스트 유지, 권한 제어를 터미널 인터랙션 없이 처리하고 싶을 때 사용합니다.

### 무엇인가

이 SDK는 Claude Code를 움직이는 것과 같은 에이전트 루프를 제공합니다. 여기에 기본 도구, 세션, 훅, 권한, MCP/커스텀 도구를 붙여서 실제 제품에 넣을 수 있게 합니다. 예전 문서에서 보이던 Claude Code SDK라는 이름은 현재 Agent SDK로 바뀌었습니다.

### 언제 쓰는가

다음이 필요할 때 적합합니다.

- 코드베이스를 읽고 수정하는 에이전트,
- 결과를 보고 다시 시도하는 작업 흐름,
- 여러 턴에 걸친 컨텍스트 유지,
- 승인 정책이나 보안 제어,
- 프로젝트 전용 도구 호출.

단순히 한 번 답만 받으면 되는 작업이라면 Agent SDK는 과할 수 있습니다.

### 멘탈 모델

에이전트 루프를 기준으로 보면 이해가 쉽습니다.

1. 프롬프트와 설정을 보냅니다.
2. Claude가 바로 답할지, 도구를 부를지 판단합니다.
3. SDK가 도구를 실행하고 결과를 다시 Claude에 돌려줍니다.
4. 세션은 이 대화 기록을 보존합니다.
5. 권한과 훅은 루프 바깥에서 행동을 통제합니다.

### 핵심 API와 패턴

- TypeScript와 Python 모두 `query(...)`가 진입점입니다.
- `allowedTools`, `disallowedTools`, `permissionMode`로 도구 범위를 조절합니다.
- `resume`, `continue: true`, `fork_session`으로 세션을 이어가거나 분기합니다.
- `mcpServers`와 `createSdkMcpServer(...)`로 커스텀 도구를 추가합니다.
- `hooks`와 `settings.json`으로 정책, 로깅, 자동화를 넣습니다.

간단한 TypeScript 예시:

```ts
import { query } from "@anthropic-ai/claude-agent-sdk";

for await (const message of query({
  prompt: "auth.ts의 버그를 찾아 고쳐줘",
  options: { allowedTools: ["Read", "Edit", "Bash"] },
})) {
  if (message.type === "result" && message.subtype === "success") {
    console.log(message.result);
  }
}
```

### 흔한 실수

- SDK를 상태 없는 프롬프트 래퍼처럼 생각하는 것
- 쓰기나 셸 권한을 열어두고 시작하는 것
- 훅이 권한 시스템 전체를 대체한다고 착각하는 것
- 세션 복원과 파일 상태 복원을 혼동하는 것

<a id="12-agent-sdk-12-권한"></a>

## 12-12. 권한

권한은 Claude가 어떤 도구를 쓸 수 있는지, 그리고 언제 사용자 승인이 필요한지를 정합니다. 파일 수정, 셸 명령, 기타 민감한 작업을 통제하는 핵심 안전장치입니다.

### 무엇인가

Agent SDK는 권한 모드, 선언형 규칙, 훅, 런타임 콜백을 함께 사용합니다. 이 조합이 도구 요청을 자동 허용할지, 차단할지, 아니면 승인을 요청할지를 결정합니다.

### 언제 쓰는가

다음이 필요할 때 사용합니다.

- 기본적으로 읽기 전용 에이전트로 두고 싶을 때,
- 신뢰 가능한 작업 공간에서 편집만 자동 승인하고 싶을 때,
- 위험한 명령을 차단하고 싶을 때,
- 계획만 만들고 실행은 막고 싶을 때,
- 사용자나 운영자 승인 흐름이 필요할 때.

### 멘탈 모델

권한 평가는 다음 순서로 흘러갑니다.

1. 훅이 도구 호출을 먼저 검사합니다.
2. `settings.json`의 `deny`, `allow`, `ask` 규칙이 적용됩니다.
3. 현재 권한 모드가 기본 동작을 정합니다.
4. 아직 결정되지 않으면 `canUseTool`이 판단합니다.

뒤 단계는 앞 단계의 명시적 거절을 무시하지 못합니다.

### 핵심 API와 패턴

- `permissionMode`를 `query(...)` 옵션으로 설정하고 스트리밍 중에도 바꿀 수 있습니다.
- `default`는 기본 승인 흐름입니다.
- `acceptEdits`는 파일 편집과 파일시스템 작업만 자동 승인합니다.
- `bypassPermissions`는 모든 프롬프트를 없애지만, 훅은 계속 실행됩니다.
- `plan`은 도구 실행을 막고 검토/제안 작업에 맞습니다.
- `allowedTools`, `disallowedTools`로 명시적인 도구 허용/차단을 할 수 있습니다.
- `settings.json`의 `allow`, `deny`, `ask` 규칙으로 선언형 정책을 둡니다.

### 흔한 실수

- `acceptEdits`가 Bash까지 자동 승인한다고 생각하는 것
- `bypassPermissions`를 통제되지 않은 환경에서 쓰는 것
- 하위 에이전트가 bypass 모드를 그대로 상속한다는 점을 놓치는 것
- plan 모드에서 실제 변경이 된다고 기대하는 것
- 훅이 권한 규칙을 대체한다고 생각하는 것

## 12-13. 플러그인

플러그인은 Claude Code 확장을 설치 가능하고 버전 관리 가능한 단위로 포장하는 배포 계층입니다.

### 무엇을 묶을 수 있는가

시스템에 따라 다음 같은 것들을 함께 패키징할 수 있습니다.

- 스킬
- 슬래시 명령
- MCP 서버 또는 커넥터
- 보조 지시문과 메타데이터

그래서 플러그인은 설치, 버전 관리, 활성화/비활성화, 팀 공유가 필요한 확장 기능에 적합한 경계가 됩니다.

### 언제 쓰는가

기능이 휴대 가능해야 하고 하나의 단위로 관리되어야 한다면 플러그인이 맞습니다. 반대로 특정 앱 내부에서만 쓰고 밖으로 공유할 필요가 없으면 호스트 직접 통합이 더 단순할 수 있습니다.

### 설계 가이드

- 플러그인 범위는 일관되게 유지합니다.
- 서로 무관한 기능을 하나에 억지로 섞지 않습니다.
- 설치, 설정, 안전 기대치를 문서화합니다.
- 버전 변경은 단순 포장 변경이 아니라 동작 변경일 수 있다고 보고 다룹니다.

### 흔한 실수

- 패키징이 필요 없는 작은 기능까지 플러그인으로 만드는 것
- 광범위한 도구 권한을 명확한 안내 없이 배포하는 것
- 검색이나 발견 시스템이 이해하기 어렵게 메타데이터를 빈약하게 두는 것

### 관련 링크

## 12-14. Python

Python은 자동화 백엔드, 평가 시스템, 리서치 도구, 운영 워크플로처럼 이미 Python 위에 있는 환경에 잘 맞습니다.

### 왜 고를까

다음이 필요하면 Python SDK가 적합합니다.

- 기존 Python 서비스와의 쉬운 통합

- 사내 자동화를 위한 익숙한 패키징

### 기본 형태

Python에서는 클라이언트 객체를 유지한 채 여러 쿼리를 보내는 구성이 흔합니다.

from claude_agent_sdk import ClaudeSDKClient

client = ClaudeSDKClient()
result = client.query(
    prompt="Summarize the recent release notes",
    allowed_tools=["Read", "WebFetch"],

print(result)

### 강점

- 오케스트레이션 중심 시스템에 잘 맞습니다.
- 평가, 분석, 노트북 워크플로와 자연스럽게 연결됩니다.
- 세션 지향 클라이언트 모델이 분명합니다.

### 흔한 실수

- Python 세션이 디스크에 저장된다는 점을 놓치는 것
- 노트북 실험을 운영 보안 경계와 같은 수준으로 생각하는 것
- 운영 작업에 너무 많은 도구를 허용하는 것

## 12-15. 보안 배포

### 핵심 통제 지점

<a id="12-agent-sdk-16-세션"></a>

## 12-16. 세션

세션은 Agent SDK가 쌓아 두는 대화 기록입니다. 프롬프트, 도구 호출, 도구 결과, 모델 응답을 함께 저장해서 나중에 같은 작업을 이어갈 수 있게 합니다.

### 무엇인가

세션은 파일 시스템 스냅샷이 아니라 대화 기록과 메타데이터입니다. SDK가 자동으로 디스크에 저장하므로, 나중에 같은 흐름으로 돌아오거나 분기할 수 있습니다.

### 언제 쓰는가

다음 상황에서 세션이 필요합니다.

- 한 프로세스 안에서 여러 턴을 이어가는 작업,
- 프로세스를 재시작한 뒤 이어서 진행해야 하는 경우,
- 같은 출발점에서 다른 접근을 시험하고 싶은 경우,
- 세션 조회, 정리, 분류가 필요한 경우.

### 멘탈 모델

세션은 “Claude가 알고 있는 것”이지 “현재 파일 상태”가 아닙니다. Claude가 이미 읽은 파일, 수행한 분석, 내린 결정은 세션에 남습니다. 반대로 파일 변경을 되돌리려면 파일 체크포인팅을 써야 합니다.

### 핵심 API와 패턴

- Python의 `ClaudeSDKClient`는 세션 ID를 내부에서 관리합니다.
- TypeScript의 `continue: true`는 현재 디렉터리에서 가장 최근 세션을 이어갑니다.
- `resume`는 특정 세션 ID를 지정해서 복원합니다.
- `fork_session: true`는 원본 기록을 복사한 새 세션을 만듭니다.
- `persistSession: false`는 TypeScript에서만 메모리 세션으로 끝냅니다.
- `ResultMessage.session_id`를 저장해 두면 나중에 안전하게 이어갈 수 있습니다.
- `listSessions()`, `getSessionMessages()`, `getSessionInfo()`, `renameSession()`, `tagSession()`로 세션을 탐색하고 정리할 수 있습니다.

### 흔한 실수

- 다른 `cwd`에서 resume하는 것
- 세션이 다른 머신으로 자동 이동한다고 기대하는 것
- fork가 파일 변경까지 되돌린다고 생각하는 것
- Python도 메모리 전용 세션을 지원한다고 착각하는 것

## 12-17. 스킬

스킬은 반복되는 작업 유형을 위해 지시문, 헬퍼 스크립트, 템플릿, 보조 자산을 함께 묶은 재사용 가능한 능력 단위입니다.

### 어떤 경우에 좋은가

다음처럼 비슷한 종류의 작업을 꾸준한 방식으로 수행하게 만들고 싶을 때 적합합니다.

- 문서 생성
- 특정 스타일의 코드 리뷰
- 릴리스 노트 준비
- 반복적인 사내 워크플로 실행

스킬은 한 줄 프롬프트보다 강합니다. 로컬 지시와 재사용 자산을 함께 운반할 수 있기 때문입니다.

### 스킬 vs 슬래시 명령

- 사용자가 이름으로 호출해야 하는 흐름이면 슬래시 명령이 맞습니다.
- 시스템이 재사용 가능한 능력과 문맥을 묶어야 한다면 스킬이 맞습니다.

실무에서는 둘을 같이 씁니다. 슬래시 명령이 워크플로를 시작하고, 스킬이 그 아래 재사용 가능한 지시를 제공하는 식입니다.

### 흔한 실수

- 하나의 스킬 폴더를 거대한 잡동사니 묶음으로 키우는 것
- 언제 발동해야 하는지 문서화하지 않는 것
- 스킬이 권한, 훅, 테스트를 대신한다고 생각하는 것

## 12-18. 슬래시 명령

슬래시 명령은 반복적인 워크플로를 이름 붙은 진입점으로 감싼 단축 명령입니다. 사용자가 긴 프롬프트를 기억하지 않아도 안정적인 호출 표면을 제공할 수 있습니다.

### 왜 중요한가

다음처럼 반복되는 흐름에 특히 잘 맞습니다.

- 코드 리뷰
- 배포 준비
- 프로젝트 전용 보고서
- 사고 대응
- 개발 환경 설정

실제로는 복잡한 로직 덩어리보다는, 잘 정의된 워크플로의 얇은 진입점으로 유지하는 편이 좋습니다.

### 설계 가이드

- 명령 이름은 짧고 기억하기 쉽게 둡니다.
- 필요한 입력과 부작용을 문서화합니다.
- 명령이 펼쳐지는 워크플로를 예측 가능하게 만듭니다.
- 조금씩만 다른 중복 명령을 남발하지 않습니다.

### 흔한 실수

- 기본 UX가 약한 문제를 슬래시 명령 남발로 덮는 것
- 거의 같은 명령을 여러 개 만드는 것
- 파괴적 동작을 무해해 보이는 이름 뒤에 숨기는 것

## 12-19. 스트리밍 출력

스트리밍 출력은 한 번에 최종 텍스트만 받는 대신, 실행 과정을 이벤트 시퀀스로 노출하는 방식입니다.

### 무엇이 오는가

실행에 따라 다음 같은 이벤트를 받을 수 있습니다.

- 부분 응답 텍스트
- 도구 요청 이벤트
- 도구 결과 이벤트
- 상태나 사용량 업데이트
- 최종 성공 또는 실패 메시지

그래서 전체 실행이 끝날 때까지 기다리지 않고도 반응형 UI와 추적 체계를 만들 수 있습니다.

### 언제 쓰는가

다음이 필요하면 스트리밍 출력이 적합합니다.

- 실시간 터미널 또는 채팅 렌더링
- 진행 표시나 타임라인
- 승인 흐름 처리
- 도구 사용 관측성
- 점진적 JSON 또는 이벤트 처리

### 호스트 책임

호스트 애플리케이션은 이벤트를 어떻게 UI와 로그로 매핑할지 결정해야 합니다. 보통은:

- 부분 텍스트를 즉시 렌더링하고
- 승인이나 사용자 입력 이벤트에서는 잠시 멈추고
- 도구 이벤트는 사용자용 텍스트와 별도로 기록하고
- 최종 결과 메시지를 완료 신호로 취급합니다

### 흔한 실수

- 모든 이벤트를 그대로 사용자에게 이어붙이는 것
- 부분 메시지를 최종 메시지처럼 다루는 것
- 도구 이벤트를 무시해서 UI가 불투명해지는 것
- 중단되거나 실패한 실행을 깔끔하게 처리하지 않는 것

<a id="12-agent-sdk-20-streaming-vs-single-mode"></a>

## 12-20. Streaming vs Single Mode

Agent SDK에는 크게 두 가지 실행 스타일이 있습니다. 이벤트를 순서대로 받는 스트리밍 모드와, 한 번 실행해서 최종 결과만 받는 싱글 모드입니다.

### 스트리밍 모드

호스트 애플리케이션이 실행 중간에도 반응해야 한다면 스트리밍 모드를 씁니다.

대표적인 경우:

- 부분 응답을 실시간으로 보여줄 때
- 도구 호출과 승인 흐름을 처리할 때
- 사용자 후속 입력을 받아야 할 때
- 사용량, 상태, 중간 이벤트를 추적할 때
- 장시간 실행되는 에이전트를 계속 반응형으로 유지할 때

실제 에이전트 앱에서는 도구와 라이프사이클 이벤트가 모두 드러나는 스트리밍이 기본 선택에 가깝습니다.

### 싱글 모드

전체 실행이 끝난 뒤 최종 결과 하나만 받으면 되는 경우 싱글 모드가 단순합니다.

대표적인 경우:

- 배치 작업
- 단순 자동화
- 구조화된 출력 파이프라인
- 중간 이벤트가 필요 없는 래퍼 서비스

다만 중간 제어와 가시성은 줄어듭니다.

### 어떻게 고를까

다음 중 하나라도 해당하면 스트리밍 모드가 더 적합합니다.

- 에이전트가 도구를 호출할 수 있다
- 사용자 승인이나 답변이 필요하다
- 실시간 출력이나 추적이 필요하다
- 세밀한 가드레일을 걸고 싶다

최종 응답만 있으면 충분하고 실행 중간이 불투명해도 괜찮다면 싱글 모드가 맞습니다.

### 흔한 실수

- 처음에 싱글 모드로 시작했다가 승인 흐름이 필요해지는 것
- 구조화된 출력이면 무조건 싱글 모드가 낫다고 생각하는 것
- 커스텀 도구나 MCP 통합도 싱글 모드에 자연스럽다고 가정하는 것

<a id="12-agent-sdk-21-구조화된-출력"></a>

## 12-21. 구조화된 출력

구조화된 출력은 자유 형식 텍스트 대신 예측 가능한 스키마에 맞는 결과를 받는 방식입니다.

### 무엇인가

"JSON으로 반환해 줘"라고 막연히 요청하는 대신, 애플리케이션이 기대하는 구조를 먼저 정합니다. 그러면 SDK가 최종 결과를 그 구조에 맞춰 검증하거나 제한할 수 있습니다.

### 언제 쓰는가

다음으로 바로 넘겨야 하는 결과라면 구조화된 출력이 유리합니다.

- 다른 서비스
- 워크플로 엔진
- 데이터베이스 기록
- 평가 하니스
- 타입이 정해진 프론트엔드 컴포넌트

싱글 모드에서 특히 많이 쓰이지만, 스트리밍 모드에서도 마지막 이벤트를 구조화된 페이로드로 다룰 수 있습니다.

### 설계 가이드

- 스키마는 좁고 목적 지향적으로 둡니다.
- 라우팅 결정에는 enum이나 제한된 필드를 선호합니다.
- 기계가 읽는 최종 결과와 에이전트의 추론 텍스트를 분리합니다.
- 다운스트림이 민감하면 호스트에서도 한 번 더 검증합니다.

### 흔한 실수

- 작업 범위보다 훨씬 넓은 스키마를 요구하는 것
- 기계용 필드 안에 설명 문장을 섞는 것
- 스키마 검증이 비즈니스 규칙 검증까지 대신한다고 생각하는 것

## 12-22. 서브에이전트

서브에이전트는 좁은 범위의 작업을 격리된 컨텍스트에서 처리한 뒤 결과를 부모 워크플로에 돌려주는 전문화된 하위 에이전트입니다.

### 왜 쓰는가

하나의 에이전트가 모든 세부사항을 한 컨텍스트에 들고 갈 필요가 없을 때 유용합니다. 책임 분리, 병렬 작업, 부모 에이전트의 조정 역할 유지에 도움이 됩니다.

좋은 예:

- 코드 탐색과 구현 분리
- 조사와 실행 분리
- 파일 범위별 소유권 분리
- 리뷰나 검증 전용 패스

### 멘탈 모델

부모는 조정자이고, 서브에이전트는 보조 작업자입니다. 중요한 경로를 무작정 떠넘기기보다는 다음 원칙을 지키는 편이 좋습니다.

- 과제를 좁게 정의한다
- 소유권을 명확히 준다
- 쓰기 범위가 겹치지 않게 한다
- 결과를 의도적으로 병합한다

### 흔한 실수

- 모호한 작업을 서브에이전트에 넘기는 것
- 당장 막힌 핵심 작업을 위임하고 기다리기만 하는 것
- 여러 에이전트를 같은 파일 집합에 보내는 것
- 병렬성의 이익만 보고 검토 비용을 무시하는 것

## 12-23. Todo Tracking

Todo tracking은 장시간 또는 다단계 작업에서 에이전트와 호스트가 공유할 수 있는 가벼운 작업 목록입니다.

### 무엇인가

자유 형식 설명만으로 진행 상황을 추측하게 두는 대신, 에이전트가 명시적인 작업 목록을 유지하고 상태를 갱신하며 무엇이 남았는지 보여줄 수 있게 합니다.

### 언제 유용한가

다음과 같은 경우에 특히 효과적입니다.

- 장시간 리팩터링
- 여러 파일을 건드는 마이그레이션
- 리뷰 워크플로
- 백그라운드 작업
- 운영자 대시보드

### 설계 가이드

좋은 todo 목록은 다음 특성을 가집니다.

- 짧다
- 결과 중심이다
- 작업 방향이 바뀌면 같이 갱신된다
- 호스트나 운영자에게 보인다

대화 전체를 또 하나의 목록으로 복제하는 수준이 되어서는 안 됩니다.

### 흔한 실수

- 실제 실행 상태와 목록이 어긋나는 것
- 체크 완료가 불가능할 만큼 모호한 항목을 쓰는 것
- 대화 전체를 todo 목록에 복사하는 것

## 12-24. Tool Search

Tool search는 사용 가능한 도구 표면이 너무 클 때, 관련 도구만 먼저 찾아서 모델에 보여주기 위한 메커니즘입니다.

### 무엇을 하는가

모든 도구를 처음부터 전부 컨텍스트에 넣는 대신, 호스트가 도구 메타데이터 검색 인덱스를 제공합니다. Claude는 먼저 그 인덱스를 검색하고, 실제로 필요한 도구만 로드하거나 사용합니다.

이 방식은 컨텍스트를 작게 유지하고 도구 선택 정확도를 높입니다.

### 언제 쓰는가

다음 조건이면 특히 유용합니다.

- 플랫폼에 도구나 커넥터가 많다
- 도구가 플러그인, 테넌트, 앱 단위로 나뉜다
- 작업마다 필요한 도구가 소수다
- 프롬프트를 영구적으로 비대하게 만들고 싶지 않다

### 설계 가이드

- 도구 설명은 구체적이고 행동 중심으로 적습니다.
- 안전과 관련된 제약도 메타데이터에 포함합니다.
- 도구가 바뀌면 검색 인덱스도 함께 갱신합니다.
- 어떤 도구가 노출됐고 최종 선택됐는지 기록합니다.

tool search는 특히 커넥터가 많은 시스템에서 유용하지만, 실행 권한 검사를 대신하지는 못합니다.

### 흔한 실수

- 메타데이터가 모호해서 잘못된 도구가 더 적합해 보이게 만드는 것
- 검색 결과가 곧 실행 권한이라고 착각하는 것
- 오래된 검색 인덱스를 방치하는 것

### 관련 링크

## 12-25. TypeScript v2 Preview

TypeScript v2 preview는 다음 세대 TypeScript SDK 표면을 다루는 문서입니다. 안정 버전이라기보다, 앞으로 옮겨 갈 방향을 보여주는 마이그레이션 대상에 가깝습니다.

- 필요한 기능이 프리뷰 쪽에 먼저 들어왔을 때
- 현재 TypeScript 패키지에서 옮겨갈 계획을 세울 때

- 도입 전에 정확한 패키지와 버전을 다시 확인합니다.

## 12-26. TypeScript

TypeScript는 호스트 애플리케이션이 이미 Node.js 위에 있거나, 이벤트와 도구, 실행 옵션을 타입으로 강하게 다루고 싶을 때 가장 빠른 경로입니다.

### 왜 고를까

다음이 필요하면 TypeScript SDK가 잘 맞습니다.

- Node 네이티브 통합
- 타입이 있는 스트리밍 이벤트 처리
- 웹 백엔드나 CLI에 자연스러운 임베딩
- 비동기 워크플로에 대한 세밀한 제어

### 기본 형태

패키지를 설치한 뒤 앱에서 `query(...)`를 호출합니다.

```ts
import { query } from "@anthropic-ai/claude-agent-sdk";

for await (const message of query({
  prompt: "Review the recent auth changes",
  options: { allowedTools: ["Read", "Grep", "Glob"] },
})) {
  if (message.type === "result") {
    console.log(message.result);
  }
}
```

### 강점

- 스트리밍과 async iteration 조합이 깔끔합니다.
- 기존 Node 서비스에 붙이기 쉽습니다.
- MCP나 커넥터가 많은 호스트에 잘 맞습니다.
- 타입이 있는 스키마와 이벤트 처리에 유리합니다.

### 흔한 실수

- 실제로는 스트리밍이 필요한데 싱글 모드로 시작하는 것
- 도구 권한을 충분히 좁히지 않는 것
- 일회성 세션 처리와 운영용 지속 세션을 섞는 것

## 12-27. 사용자 입력

Agent SDK에서 사용자 입력은 에이전트가 중간에 멈춰서 설명, 승인, 구체적인 값을 사람에게 요청한 뒤 같은 실행을 이어가는 흐름을 뜻합니다.

### 무엇인가

실제 에이전트 앱에서는 파일이나 도구만으로는 알 수 없는 정보가 자주 나옵니다. 이때 Claude가 추측하지 않고 호스트에게 질문을 올리고, 답을 받은 뒤 같은 실행 컨텍스트에서 이어서 진행할 수 있습니다.

### 좋은 사용 사례

- 여러 선택지 중 무엇을 고를지 사용자에게 묻는 경우
- 프롬프트에 직접 넣기 어려운 ID나 비밀 값을 받는 경우
- 파괴적 작업 전에 최종 승인을 받는 경우
- 모호한 요청을 좁힌 뒤 추가 도구 실행을 하고 싶은 경우

### 설계 가이드

입력 요청은 짧고 명확해야 합니다. 호스트 UI는 다음을 분명히 보여줘야 합니다.

- Claude가 무엇을 묻는지
- 왜 그 답이 필요한지
- 그 답이 권한, 비용, 범위를 바꾸는지

위험한 답변이라면 자유 입력창 안에 숨기지 말고 UI 차원에서 위험을 드러내는 편이 좋습니다.

### 흔한 실수

- 에이전트가 너무 모호한 질문을 던져 작업을 멈추게 만드는 것
- 승인 요청과 일반 확인 질문을 섞는 것
- 사용자가 답한 뒤 세션 상태를 제대로 이어 붙이지 않는 것
- 기본값 설계 대신 사용자 입력에 과도하게 의존하는 것
