# 위키독스 업로드용 작성 양식 가이드

기준일: 2026-04-24

## 목적

이 프로젝트의 원문은 웹앱형 문서 구조를 전제로 작성되어 있습니다.  
위키독스에서는 "브라우저에서 순서대로 읽는 책"처럼 보여야 하므로, 문서를 작성할 때도 탐색형 문서보다 독서형 문서 형식을 우선하는 편이 낫습니다.

핵심 원칙은 다음 4가지입니다.

1. 한 페이지에서 주제를 끝까지 이해할 수 있게 쓴다.
2. 링크로 흩어 보내기보다, 먼저 핵심 설명을 본문에 담는다.
3. 독자가 "언제 쓰는지"를 빠르게 알 수 있게 시작한다.
4. 실전 예시와 주의사항을 붙여 바로 써먹을 수 있게 한다.

## 추천 페이지 유형

위키독스용 문서는 아래 3가지 유형으로 나누는 게 가장 안정적입니다.

### 1. 모듈 메인 페이지

예: `01-slash-commands.md`, `05-mcp.md`

역할:

- 해당 주제의 전체 지도
- 언제 쓰는지
- 하위 개념 요약
- 추천 읽기 순서

추천 구성:

1. 한 문단 요약
2. 언제 쓰나
3. 핵심 개념 표
4. 빠른 시작 예시
5. 자주 하는 실수
6. 관련 명령/설정/링크

### 2. 레퍼런스 페이지

예: 향후 다시 분리할 `analytics`, `auto-mode-config`, `plugin-dependencies`

역할:

- 설정값, 옵션, 명령, 제한사항 정리

추천 구성:

1. 무엇을 설정하는 문서인가
2. 기본값 / 권장값
3. 예시 설정
4. 자주 보는 오류
5. 운영 팁

### 3. 부록 페이지

예: Anthropic blog 기반 요약

역할:

- 공식 문서의 기능 설명이 아니라 운영 철학, 사례, 베스트 프랙티스 정리

추천 구성:

1. 글의 핵심 메시지
2. 실무적으로 중요한 포인트 3~5개
3. 우리 책의 어떤 장과 연결되는지
4. 바로 적용할 체크리스트

## 추천 문체

위키독스에서는 아래 형식이 잘 맞습니다.

- 첫 3문장에서 주제, 용도, 독자 이득을 분명히 적기
- "정의"보다 "언제 쓰는지"를 먼저 적기
- 긴 배경 설명보다 짧은 단락 여러 개로 끊기
- 표는 개념 비교, 옵션 비교, 모드 비교에만 쓰기
- 예시는 바로 복붙 가능한 명령 위주로 쓰기

피하는 편이 좋은 것:

- 웹앱 탐색 전제를 깔고 "다음 페이지로 가면 된다" 식으로 넘기기
- 너무 긴 링크 모음
- 소개 페이지인데 레퍼런스 세부사항을 과도하게 다 넣기
- 반대로 레퍼런스 페이지인데 개념 설명 없이 옵션만 나열하기

## 모듈 메인 페이지 권장 템플릿

```md
# 09. 고급 기능

이 장은 Claude Code를 단순 채팅 도구가 아니라 자율 실행 환경으로 사용할 때 필요한 기능을 정리합니다.  
권한 모드, auto mode, remote control, 웹 세션, 음성 입력, 상태 줄처럼 실제 사용감에 큰 영향을 주는 기능이 중심입니다.

## 언제 읽으면 좋은가

- 기본 명령과 메모리, 스킬, MCP를 이미 한 번 써봤을 때
- 장시간 작업이나 자동화를 안정적으로 굴리고 싶을 때
- 팀 환경에서 설정, 보안, 운영 감각까지 함께 잡고 싶을 때

## 이 장의 핵심 주제

| 주제 | 왜 중요한가 |
| --- | --- |
| 권한 모드 | Claude의 자율성을 어디까지 허용할지 결정 |
| Auto mode | 승인 피로를 줄이면서 자동화를 늘림 |
| Remote Control | 로컬 세션을 다른 기기에서 이어서 작업 |
| Voice / Keybindings / Statusline | 실제 사용 속도와 조작감을 바꿈 |

## 먼저 익힐 것

1. 권한 모드와 sandbox의 차이
2. auto mode를 어디까지 신뢰할지
3. remote control과 web의 실행 위치 차이

## 빠른 시작 예시

```sh
/permissions
/sandbox
claude --permission-mode auto
claude remote-control
```

## 자주 하는 실수

- auto mode와 bypass permissions를 같은 것으로 이해하기
- remote control과 web 세션의 실행 위치를 혼동하기
- statusline, keybindings를 단순 꾸미기로 보고 넘기기

## 관련 장

- [Slash Command](01-slash-commands.md)
- [Hooks](06-hooks.md)
- [Deployment Administration](11-deployment-admin.md)
```

## 레퍼런스 페이지 권장 템플릿

```md
# Auto mode 설정

이 문서는 auto mode 분류기가 무엇을 신뢰하고 무엇을 막는지 조정하는 방법을 설명합니다.  
기본 동작만 이해하는 문서가 아니라, 실제 조직 환경에 맞게 `autoMode` 설정을 튜닝하는 문서입니다.

## 언제 필요한가

- auto mode가 내부 리포지토리 push를 계속 막을 때
- 팀 공용 버킷, 도메인, 레지스트리를 신뢰 대상으로 추가해야 할 때
- 관리형 설정으로 조직 공통 정책을 배포할 때

## 핵심 설정

| 키 | 역할 |
| --- | --- |
| `autoMode.environment` | 신뢰하는 리포지토리/도메인/버킷 정의 |
| `autoMode.allow` | 기본 차단 규칙의 예외 허용 |
| `autoMode.soft_deny` | 더 보수적으로 막고 싶은 항목 추가 |

## 최소 예시

```json
{
  "autoMode": {
    "environment": {
      "git": ["github.com/my-org/*"],
      "domains": ["api.mycompany.com"]
    }
  }
}
```

## 확인 방법

```sh
claude auto-mode defaults
claude auto-mode config
claude auto-mode critique
```

## 자주 보는 문제

- 설정을 `.claude/settings.json`에 넣었는데 공유 프로젝트 정책으로 착각
- `permissions.deny`와 `autoMode.soft_deny`의 우선순위를 혼동
- 허용 규칙을 너무 넓게 잡아 사실상 안전 장치를 무력화
```

## 부록 페이지 권장 템플릿

```md
# Anthropic 팀은 Claude Code를 어떻게 쓰는가

이 글은 Anthropic 내부 팀 사례를 바탕으로 Claude Code의 실전 운영 패턴을 정리한 부록입니다.  
기능 자체를 설명하기보다, 어떤 팀이 어떤 문제를 어떻게 해결했는지에 초점을 둡니다.

## 핵심 메시지

- Claude Code는 개발자만 쓰는 도구가 아니다.
- 병렬 세션과 문서화 루프가 생산성 차이를 만든다.
- 좋은 `CLAUDE.md`와 좋은 운영 습관이 성능 차이로 이어진다.

## 실무적으로 중요한 포인트

1. 새 팀원 온보딩에 코드 탐색보다 Claude Code를 먼저 쓴다.
2. 세션 종료 후 `CLAUDE.md` 개선 루프를 만든다.
3. PR/CI/문서 업데이트를 Claude Code 자동화 흐름에 넣는다.

## 이 책과 연결되는 장

- Memory
- Subagents
- Hooks
- GitHub Actions
```

## 지금 프로젝트에 바로 적용할 제안

### 메인 모듈 페이지

각 모듈 메인 페이지는 아래 순서를 기본형으로 맞추는 게 좋습니다.

1. 주제 설명
2. 언제 쓰는가
3. 핵심 개념/기능 표
4. 빠른 시작
5. 자주 하는 실수
6. 관련 장

### 운영/설정 문서

`11-deployment-admin`, `09-advanced-features`, `07-plugins`는 설명 순서를 아래처럼 맞추는 편이 읽기 좋습니다.

1. 왜 필요한가
2. 어떤 선택지가 있는가
3. 최소 설정 예시
4. 팀 배포 시 주의점
5. 자주 보는 오류

### 부록

Anthropic Blog 기반 글은 기능 설명 본문에 섞지 말고, `13. 부록`으로 빼는 편이 책의 흐름을 덜 깨뜨립니다.

추천 부록 묶음:

- Anthropic의 Claude Code 실전 활용
- Claude Code 자율성과 보안 운영 원칙
- Agent skills / context engineering / multi-agent 패턴
- Routines / Code Review / Web 운영 사례

## 결론

위키독스에서는 "한 페이지를 열었을 때 끝까지 읽고 바로 써볼 수 있는가"가 제일 중요합니다.  
따라서 현재 프로젝트도 웹앱형 문서 구조를 그대로 옮기기보다, 메인 페이지는 설명형으로, 세부 페이지는 레퍼런스형으로, Anthropic 블로그 내용은 부록형으로 분리하는 방식이 가장 적합합니다.
