# Authentication and IAM

This page covers how Claude Code users and automation authenticate, how organization roles affect access, and how to choose the right credential path for interactive use, CI, and service-style deployments.

## Login Paths

For interactive use, the normal entry point is to run `claude` and complete the browser-based login flow. From there, Claude Code supports these account paths:

- Claude.ai accounts for Claude Pro or Max.
- Claude for Teams or Claude for Enterprise accounts.
- Claude Console accounts for API-based billing.
- Cloud provider auth for Amazon Bedrock, Google Vertex AI, or Microsoft Foundry.

Common shell actions:

```bash
claude
```

If you need to log out, use:

```text
/logout
```

Claude Code also supports `/login` and `/status` inside the CLI for re-authentication and verification.

## Organization And Roles

For team setups, Claude for Teams or Claude for Enterprise is the cleanest path when you want centralized billing and shared org management.

For Claude Console-based access, the important roles are:

- `Claude Code` role: users can only create Claude Code API keys.
- `Developer` role: users can create any kind of API key.

That split matters when you want to limit key creation to Claude Code use only.

## Credential Precedence

Claude Code resolves credentials in a strict order when multiple sources are present:

1. Cloud provider credentials, when `CLAUDE_CODE_USE_BEDROCK`, `CLAUDE_CODE_USE_VERTEX`, or `CLAUDE_CODE_USE_FOUNDRY` is set.
2. `ANTHROPIC_AUTH_TOKEN`, sent as `Authorization: Bearer`.
3. `ANTHROPIC_API_KEY`, sent as `X-Api-Key`.
4. `apiKeyHelper` output.
5. `CLAUDE_CODE_OAUTH_TOKEN`, the long-lived OAuth token generated by `claude setup-token`.
6. Subscription OAuth credentials from `/login`.

Two practical consequences:

- If `ANTHROPIC_API_KEY` is set and approved, it overrides a logged-in subscription.
- `ANTHROPIC_AUTH_TOKEN` takes priority over the API key and is the right choice for bearer-token gateways or proxies.

`apiKeyHelper`, `ANTHROPIC_API_KEY`, and `ANTHROPIC_AUTH_TOKEN` apply to terminal CLI sessions only. Claude Desktop and remote sessions use OAuth.

## Credential Storage

Claude Code stores credentials securely:

- macOS: encrypted Keychain.
- Linux and Windows: `~/.claude/.credentials.json`, or `$CLAUDE_CONFIG_DIR` if set.

The docs also note that Linux writes the file with `0600` permissions and Windows inherits your profile's access controls.

## CI And Service-Style Usage

Use a non-interactive credential path when browser login is not available:

- For subscription-based automation, generate a long-lived token with `claude setup-token` and export it as `CLAUDE_CODE_OAUTH_TOKEN`.
- For API-key-based pipelines, store `ANTHROPIC_API_KEY` as a masked secret.
- For cloud-provider-backed automation, use the provider's native auth model instead of static keys.

Examples of service-style patterns:

- Google Vertex AI with Workload Identity Federation and a dedicated service account.
- AWS Bedrock with OIDC-based role assumption.
- GitLab CI/CD or similar runners with masked variables and least-privilege cloud roles.

For GitLab-style automation, Anthropic's docs recommend using a masked `ANTHROPIC_API_KEY` for the Claude API path or cloud-provider configuration for Bedrock and Vertex. For Vertex specifically, the preferred pattern is a dedicated service account with only the required roles and Workload Identity Federation rather than downloadable keys.

## Practical Verification

Check which identity is active:

```text
/status
```

If you are troubleshooting a stale key path, clear the override and restart:

```bash
unset ANTHROPIC_API_KEY
claude
```

For CI or scripted environments, verify that the long-lived token or provider credentials are present before launch.
